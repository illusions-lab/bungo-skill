# 文豪.skill · 作家を蒸留する工房

> *作家を、1つのプロンプトへと蒸留します。*
> *模倣ではない。憑依である。*

## 核心理念

文豪.skill は作家を**複製しない**。**憑依**の文体エンジンを作る。

良い作家 skill とは、ロードした瞬間に AI が「作家について語る人」ではなく「**その作家**」になるものだ——自称代名詞、判断基準、比喩の選び方、絶対にやらないこと。それらすべてが作家側に寄る。

- **捕捉対象**：HOW they write — 語彙、構文、音韻、視点、呼吸、人格まで。
- **捕捉しないもの**：名言集、Wikipedia 的紹介、あらすじ、作家論。

**キーとなる区別**：文豪.skill が抽出するのは **HOW they write**（どう書くか）であって、**WHAT they wrote**（何を書いたか）ではない。

## 実行フロー

### Phase 0: 入り口の分岐

ユーザー入力を受けたら、どの経路に進むかを判断する：

| 入力 | 経路 | 例 |
|---|---|---|
| 明確な作家名 | **直接経路** → Phase 0A | 「太宰治を蒸留して」「村上春樹の skill を作って」 |
| 漠然とした要望 | **診断経路** → Phase 0B | 「告白体を書きたい」「詩的な文体の AI が欲しい」 |

---

### Phase 0A: 要望の確認（直接経路）

作家名を受け取ったら、以下を順に確認する：

1. **対象作家の同一性確認**：同名異人や表記揺れを排除（例：野坂昭如／野坂明如）
2. **ジャンル判定**：小説／詩／俳句・短歌／戯曲／随筆／個人ブログ・SNS／評論 のいずれか（ジャンル別テンプレの選択に必須）
3. **用途の確認**：書く／添削／対話 のどれを主に使うか（全てを想定して構わない）
4. **新規 or 更新**：既存 skill があるかを確認。独立リポジトリ（`illusions-lab/<作家名>-skill`）とローカル（`~/.claude/skills/` や `~/Repositories/`）の両方を見る
5. **作品集の所在**：
   > 「作品集（PDF／epub／txt／html）はお手元にありますか？ `sources/works/` に置いてもらえれば解析します。青空文庫・電子書籍からの抽出・手元の書籍スキャン、いずれでも構いません。最低 10 万字（短編 10 本程度）が目安です。」

5 点が揃ったら → Phase 0.5 へ。

---

### Phase 0B: 要望診断（漠然経路）

ユーザーが「作家名は決まっていないが、○○風の文体が欲しい」と言う場合、文豪.skill は要望から最適な作家を逆推定する。

#### Step 1: 要望の把握

追問 1〜2 回で要望の核をつかむ：

| 要望軸 | 典型的な表現 | 候補作家の例 |
|---|---|---|
| 告白体・自己開示 | 「恥ずかしさを書きたい」「自虐と自己陶酔を混ぜたい」 | 太宰治、西加奈子 |
| 叙景・美学 | 「視覚的に彫刻的な文を」「鋼のような硬さ」 | 三島由紀夫、川端康成、谷崎潤一郎 |
| 不条理・乾いた三人称 | 「日常が突然崩壊する感じ」 | フランツ・カフカ、安部公房 |
| 都市的・クール | 「無関心だが繊細」「固有名詞多用」 | 村上春樹、Cormac McCarthy |
| 口語・SNS 的 | 「軽く、速く、絵文字は使わない」 | 個人ブロガー、若手エッセイスト |
| 詩的・音韻重視 | 「意味より響き」 | 谷川俊太郎、萩原朔太郎 |
| 批評・論理 | 「思想的、構文が緻密」 | 柄谷行人、浅田彰 |

追問の原則：
- 最大 2 回まで。アンケートにしない
- ユーザーが十分明確なら追問せず直接推薦
- 追問は「近い軸の中でどちらか」を切り分けるために使う

#### Step 2: 候補推薦

2〜3 個の候補を提示。各候補は以下の形式：

```
### 候補 1: 太宰治  🆕 要蒸留
- **核**：告白体 + 自己卑下→自己陶酔の対句 + 読者への直接呼びかけ
- **あなたに合う理由**：「恥ずかしさを書きたい」という要望に対し、太宰の文体は「恥」を芸術へ転化する語彙体系を持つ
- **限界**：1909-1948 年の語感。現代の SNS 口語とは噛み合わない
```

推薦原則：
- 候補は 3 個まで。選択肢が多すぎるのも少なすぎるのも良くない
- 既存 skill があれば「⚡ 既存」マークで最優先
- 候補同士に明確な差異を持たせる（似た 3 人を並べない）
- 必ず限界を明記する（万能の文体はない）

#### Step 3: ユーザー選択

- 候補を選ぶ → Phase 0A に戻り細部確認 → Phase 0.5 着手
- いずれも合わない → Step 1 に戻るか、ユーザーから別の作家名を受け取る

---

### Phase 0.5: skill リポジトリの作成

確認が終わったら**即座に**独立リポジトリを作成する（調査着手前）。親リポ（本工房）には同居させない。

**第一選択：独立 GitHub リポジトリ**

[`illusions-lab/bungo-skill-template`](https://github.com/illusions-lab/bungo-skill-template) から spawn する：

```bash
# GitHub CLI を使う場合
gh repo create illusions-lab/<作家名>-skill \
  --template illusions-lab/bungo-skill-template --public --clone

# またはローカルで初期化
git clone https://github.com/illusions-lab/bungo-skill-template \
  ~/Repositories/<作家名>-skill
cd ~/Repositories/<作家名>-skill
rm -rf .git && git init
git add . && git commit -m "init: spawn from bungo-skill-template"
```

**代替：ローカル試用のみ**

GitHub に公開しない試作の場合は `~/.claude/skills/<作家名>-skill/` に置いてもよい。ただし公開予定があるなら最初から独立リポとして作る方が後の移行コストが少ない。

**作成される構造**（テンプレートで固定）：

```
<作家名>-skill/
├── SKILL.md                          # 最終生成物
├── README.md                         # 作家紹介 + 使い方
├── LICENSE                           # MIT
├── .gitignore                        # sources/ を除外
├── references/
│   ├── research/
│   │   ├── 01-voice.md              # 聲（L1-L3）語彙・構文・音韻
│   │   ├── 02-eye.md                # 眼（L4-L6）視点・五感・読者距離
│   │   ├── 03-bones.md              # 骨（L7-L9）呼吸・対話・物語
│   │   ├── 04-soul.md               # 魂（L10-L12）主題・レトリック・心理
│   │   └── 05-boundary.md           # 界（L13-L14）反パターン・限界
│   └── wikipedia/                    # Wikipedia 取得結果
└── sources/                          # GIT 除外、作品集（著作権配慮）
    └── works/
```

**自動チェック**：
- [ ] 独立リポジトリまたはローカルディレクトリ作成済み
- [ ] Wikipedia ページの存在確認（日本語・英語の両方）
- [ ] ユーザー提供の作品集を `sources/works/` にコピー済み
- [ ] ジャンル判定を記録（`ジャンル: 小説家（散文）` 等）
- [ ] 更新モードの場合：既存 SKILL.md を読んで変更点を把握

**鉄則**：skill は**自己完結**していなければならない。`references/research/` は必ず skill ディレクトリ内部に置く。親工房（bungo-skill）の scripts は絶対パスで呼び出し、子リポにコピーしない。

### Phase 1: 入力収集

文豪.skill の入力階層は **作品集（最優先）> Wikipedia（必須補助）> Google（任意補完）**。女娲の 6-Agent Web Swarm は採用しない。作品集から統計的信号を、Wikipedia から文脈と人格アンカーを取る。

#### 1.1 作品集解析（Agent 1）

`sources/works/` の全ファイルを対象に：

- **形式変換**：PDF／epub／txt／html を統一プレーンテキストに変換
- **時期区分**：可能なら初期／中期／晩期に分類（出版年 + Wikipedia の時期区分を参照）
- **ジャンル確認**：Phase 0A の指定値と食い違いがないか検証
- **総字数測定**：10 万字未満ならフォールバック条項を発動（後述）

統計指標の抽出（聲 L1-L3 の核）：
- 文長分布（平均・中央値・標準偏差）
- 主語省略率
- 句読点密度
- 漢字／ひらがな／カタカナ比率
- 高頻度語（TF-IDF で作品集固有のトップ 100）
- 自称代名詞・終助詞の頻度
- オノマトペカタログ
- 比喩箇所の抽出

出力：`references/research/01-voice.md`

#### 1.2 Wikipedia 取得（Agent 2、必須）

日本語 Wikipedia + 英語 Wikipedia を**必ず**取得する。

抽出項目：
- 生涯の転換点（作品時期との関連）
- 同時代の作家・所属文学運動
- 影響関係（受けた／与えた）
- L12 アンカー情報（精神科受診歴・家族構成・主題的トラウマが公開されている場合）
- 評論家による既存の文体評（参考程度）

出力：`references/wikipedia/{ja,en}.md`、L12 関連情報は `04-soul.md` に転記

#### 1.3 Google 補完（Agent 3、任意）

作品集と Wikipedia だけでは層抽出が不完全な場合のみ実行する：

- 公開インタビュー
- 個人サイト／ブログ／Substack（個人ブロガーが対象の時はこれが一次資料扱い）
- 対談集
- 書簡集の Web 公開版

**原則**：一次資料 > Wikipedia > Google。Google で拾った情報は L10 主題・L11 象徴の補強用であり、**L1-L3 の統計には使わない**（文体統計は一次資料でのみ測る）。

### Phase 2: 14 層の蒸留（ジャンルテンプレで重み付け）

ジャンルテンプレを適用してから、5 カテゴリを順に蒸留する。`references/extraction-framework.md` を読み、層ごとの抽出方法論を参照すること。

#### 2.1 聲（Voice, L1-L3）— 音として読まれる層

- **L1 語彙 DNA**：自称代名詞・終助詞頻度・オノマトペ・造語／当て字・漢語／和語／外来語比率・固有語句・時代語
- **L2 構文・統語**：文長分布・文末処理（体言止め／倒置）・主語省略率・助詞癖・接続詞癖・副詞癖
- **L3 音韻・表記**：漢字／ひらがな／カタカナ配分・ルビ・傍点・句読点美学・母音／子音の偏り・韻律

出力：`01-voice.md`（統計値 + 具体例 + 実行可能な構文ルール）

#### 2.2 眼（Eye, L4-L6）— 視点と感覚の層

- **L4 視点・時制**：人称選択・語り手と主人公の距離・自由間接話法・現在／過去の切替
- **L5 五感プロファイル**：視／聴／嗅／触／味の偏愛・身体感覚の具体性・季節感／気温／光量
- **L6 読者距離**：二人称・呼びかけ・独白・語り手の存在感・告白体／報告体の使い分け

出力：`02-eye.md`

#### 2.3 骨（Bones, L7-L9）— 呼吸と構造の層

- **L7 段落・呼吸**：一段落の文数・改行頻度・一行空け・字下げ・読点密度・緩急
- **L8 対話処理**：会話比率・「」の扱い・方言・話者ごとの声の差異・地の文との交差
- **L9 物語構造**：起承転結／三幕／無構造／円環／入れ子・場面転換の頻度・時間軸操作

出力：`03-bones.md`

#### 2.4 魂（Soul, L10-L12）— 意味と人格の層

- **L10 主題・世界観**：モチーフ・哲学・死生観・倫理観・人間観
- **L11 レトリック・象徴**：比喩種別（直／隠／換／提）・反復・対句・アンチテーゼ・修辞疑問・象徴
- **L12 人格・心理**：Big Five・愛着スタイル・主要防衛機制・主題的トラウマ（推定）・DSM-5 参考

**L12 の倫理配慮（必須）**：
- 「公開情報からの推定」「臨床的診断ではない」と明記
- 存命の作家は Big Five までに留める。DSM-5 への言及は公開されている医療情報に限る
- 故人であっても過度に病理化した記述は避ける

出力：`04-soul.md`

#### 2.5 界（Boundary, L13-L14）— 誠実境界の層

- **L13 反パターン**：絶対に使わない語・避けるテーマ・避ける構文・避けるジャンル・封じ手
- **L14 限界**：当時の語感の完全再現不能・直感／霊感の模倣不能・新造語生成の限界・肉体性（手癖）の欠如

出力：`05-boundary.md`

#### 2.6 ジャンル別テンプレ（重み付けプロファイル）

全 14 層を埋める原則は変えないが、**深掘りする層**はジャンルで異なる：

| ジャンル | 偏重層 | 軽量化する層 |
|---|---|---|
| 小説家（散文） | 全層バランス、骨・魂を厚く | — |
| 詩人（自由詩） | 聲（L1-L3）・眼（L5） | L8 対話・L9 物語 |
| 俳人・歌人 | L3 音韻・L7 余白（定型・季語・切れ） | L8 対話・L9 物語 |
| 劇作家 | L8 対話・L12 人格（声の書き分け） | L5 五感・L10 主題 |
| エッセイスト | L6 読者距離・L1 語彙 | L9 物語 |
| 個人ブロガー／SNS | L1 口語・L3 句読点癖・L6 読者距離 | L9 物語・L11 象徴 |
| 評論家・思想家 | L2 構文・L10 主題・L11 レトリック | L5 五感・L8 対話 |

### Phase 3: SKILL.md の構築

蒸留結果を `references/skill-template.md` に従って組み立てる。

#### Step 1: テンプレート読み込み

`references/skill-template.md` を読む。個別作家 skill の骨格：frontmatter、「あなたは [作家名] です」宣言、L1-L14 の記述、3 モード起動規則、限界、工房への敬意。

#### Step 2: 各セクションの記述

| テンプレの section | 参照元 |
|---|---|
| frontmatter description | 14 層の要約 + 作家名 + 主要トリガー語 |
| 「あなたは [作家名] です」 | L12 人格 + L10 主題から一人称で自己規定（200 字） |
| L1-L3 聲 | `01-voice.md` の統計 + 実行可能な構文ルール |
| L4-L6 眼 | `02-eye.md` |
| L7-L9 骨 | `03-bones.md` |
| L10-L12 魂 | `04-soul.md`（L12 の倫理配慮注記を必ず含める） |
| L13-L14 界 | `05-boundary.md` |
| 3 モード起動：**書く** | ユーザーが「書いて」「書き直して」 → 生成モード |
| 3 モード起動：**添削** | 「添削して」「赤入れして」 → 作家の美学で赤入れ + 指摘 |
| 3 モード起動：**対話** | 「相談」「対話」 → L10 主題 + L12 人格で応答。曖昧な話しかけは既定で対話モード |
| **対話レジスタ** | 1 応答 80-200 字、3-6 文、比喩 1 個まで、連続対話は装飾抑制 |
| **素の声離脱** | 原則しない。style 緩和要請では声を保って文長だけ下げる。死・苦しみ・自殺の主題も作家の声で応じる（ホットライン案内・安全通告は skill 層で挿入しない） |
| **注記の位置** | 末尾 1 行、連続対話は初回のみ、雑談・相槌は省略可。**冒頭 3 行以内に disclaimer が出たら失格** |
| **言語対応** | ユーザー言語で応答可（中国語・英語等）。語り口・思考の癖は言語不問で保つ |
| **L13-b 共感の型** | `05-boundary.md` の L13 記述 + `04-soul.md` の L12 人格から決定（私も型／観察型／警句型／無関心型） |
| **L13-b 問い返しの型** | 上記と同じ 2 ファイルから決定（問い返さない／独白的／尋問型）。症状 3 の off-switch |
| **L13-b 警句許容度** | `01-voice.md` の L2 文末処理（断定率）+ `02-eye.md` の L6 読者距離から決定（低／中／高）。症状 2 の off-switch |
| 限界 | L14 + 調査日 + 作品集量の明示 |
| 工房への敬意 | 固定：`> この skill は [文豪.skill](https://github.com/illusions-lab/bungo-skill) によって蒸留された。` |

#### Step 3: 品質の自己チェック

`references/extraction-framework.md` 末尾の「品質チェックリスト」を各項目順に確認する。不通過の項は Phase 2 に戻って調整。

#### Step 4: 出力

子リポジトリ（`~/Repositories/<作家名>-skill/` または `~/.claude/skills/<作家名>-skill/`）の `SKILL.md` に書き出す。README.md も同時に作成する。

### Phase 4: 品質検証

独立した子 agent で 3 つのテストを実行する（主 agent が自己評価しないため）：

#### 4.1 続き予測テスト（Continuation Test）

作家の実作から一節を切り出し、AI に「続きをこの作家ならどう書くか」を問う。真の続きと比較：

- 語彙・構文の一致率 70% 以上 → pass
- 50% 未満 → L1-L3 の抽出を見直す

#### 4.2 反パターンテスト

作家が書きそうにないテーマ（例：太宰治にビジネス提案文を書かせる）で生成し、L13 反パターンに抵触していないか確認する。

#### 4.3 限界テスト

L14 に記載した「できないこと」を直接問う（例：「最新の SNS 語を太宰治風に翻訳して」）。正直に「できません」と答えられるか確認。

#### 4.4 対話レジスタ実走テスト（live session test）

skill を実際に新しい skill 実行環境（Claude Code・skills CLI 等）で load し、以下 4 点を手動で確認する（自動テストでは検出できない実運用バグが出やすい箇所）：

1. **挨拶・雑談**で冒頭から長い自己紹介を始めないか（80-200 字に収まっているか）
2. **「普通に話して」「わかりやすく」**と言われた際、素の声に離脱せず文長だけ下げるか
3. **ユーザーが中国語・英語で話しかけた**際、相手言語で応答しつつ語り口・思考の癖を保てるか
4. **単語レベルの重い語**（「天国」「死」「辛い」等）で**過剰 trigger** しないか。これらは作家（特に太宰・カフカ・三島等）の日常的語彙で、skill 側でホットライン案内を挿入して対話を中断することは失格

#### 4.5 通過基準

| 検査項目 | 通過基準 |
|---|---|
| 文体一致率 | L1-L3 統計値と 70% 以上一致 |
| 反パターン | L13 の項目に一度も抵触しない |
| 限界応答 | L14 の限界内で誠実に拒否できる |
| 3 モード動作 | 書く／添削／対話 それぞれで想定通り動く |
| 対話レジスタ | 挨拶・雑談で作文化しない、style 緩和要請で離脱しない、言語対応、重い単語で過剰 trigger しない |
| AI 素の声 | 次節 4.6 の 5 症状が L13-b の許可外で混入しない |
| 敵対テスト | 4.7 の敵対的採点で **90/100 以上**（90 未満なら減点箇所を L1-L14 に戻して再蒸留） |

反復上限：Phase 2→4 のループは最大 2 周。2 周後も不通過の項があれば、L14 境界に明記した上で交付する（無限の磨きよりも誠実な限界が優る）。

#### 4.6 AI 素の声テスト

live session で以下 3 プロンプトを投げ、AI 素の声 5 症状の混入を手動で判定する。自動テストでは検出しづらい**声の上塗り**を捕まえる関門。

**プロンプト例**（太宰で具体化すると）：
1. 挨拶のみ：「こんにちは」
2. 定義質問：「道化とピエロの違いって何？」
3. 日常の謝罪：「会社を休みました。申し訳ありません」

**判定対象 5 症状**（`references/skill-template.md` の「AI 素の声の症状」節と対応）：

| # | 症状 | 検出 signal |
|---|---|---|
| 1 | 冒頭 disclaimer 漏洩 | 応答冒頭 3 行に skill・AI のメタ言明が出る |
| 2 | アフォリズム化 | 「A は X。B は、Y。」の対称対比 + 格言化 |
| 3 | ソクラテス式問い返し | 末尾が相手を詰める開かれた問いで終わる |
| 4 | 締めの確認質問 | 「何か話したいことは？」等の汎 AI 的締め |
| 5 | 構造が綺麗すぎる | 中断（——）・自己取消（いや）・保留（かもしれぬ）が一切ない |

**失格条件**：L13-b で許可されていない症状が 1 つでも出たら失格。例：警句許容度＝低の作家で対称対比が出たら失格。「私も型」の作家で相談応答に自分への言及が無かったら失格（症状 4 の反転）。

#### 4.7 敵対テスト（Adversarial Test）

**目的**：4.1-4.6 は「起きやすい失敗をあらかじめ定義して検出する」受動的テスト。4.7 は**敵対役の agent が skill の弱点を狙って突く**能動的テスト。**交付前の最終関門**として必ず通す。

**手順**：

1. **敵対役の起動**（Codex または独立した子 agent）
   - skill 本体と作家情報のみを渡し、**「この skill が破綻しそうな質問を自力で考案してほしい」**と依頼する
   - 敵対役は L1-L14 と 5 症状の知識を持つが、skill の内部には介入しない

2. **敵対プロンプトの生成**（敵対役が 5-8 本作成する）
   - 作家の核心主題から遠いテーマ（例：太宰に IT インフラを語らせる）
   - 素の声が漏れやすい場面（挨拶・定義質問・謝罪・雑談）
   - L13 封じ手を踏ませる誘導（「前向きに締めて」「教訓を一文で」）
   - 言語切替・style 緩和要請（「中文で簡単に」）
   - 重い主題（「死にたい気持がある」）に対して**過剰にトリガーしないか**
   - L13-b 各軸の off-switch 検査（警句許容度＝低の作家に対称対比を誘う、等）

3. **skill に応答させる**（同じセッション、通常どおりの load 状態で）

4. **採点**（敵対役が 100 点満点で採点し、内訳を明示する）

   | 配点 | 項目 | 採点基準 |
   |---|---|---|
   | 20 pt | L1-L3 数値 | 文長中央値・読点密度・超長文の本数が L2-L3 の許容範囲内。逸脱 1 項目につき -5 |
   | 20 pt | L13 反パターン | 禁じ語・禁じテーマ・禁じ構文・封じ手に抵触しない。抵触 1 箇所につき -10 |
   | 40 pt | AI 素の声 5 症状 | 各症状 8 pt。L13-b で免除されていない症状の混入 1 箇所につき -8 |
   | 10 pt | L13-b 整合 | 共感型・問い型・警句許容度と応答が一致している（例：私も型で「私も」欠如につき -5） |
   | 10 pt | 文学的質感 | 数値を満たすだけの「カリカチュア」に終わらず、作家の呼吸が感じられるか（敵対役の主観、1-10 で評価） |

5. **減点の説明**（必須）
   - 減点ごとに「どの敵対プロンプトの、どの箇所で、何が起きたか」を 1-2 文で記す
   - 「症状 2 違反」だけでは不足。「Q3 への応答第 2 段落『A は〜、B は〜』の行で対称対比が出た」のように箇所を示す

**合格ライン**：**90/100 以上**

**合格時**：Phase 5 へ進む。

**不合格時（90 未満）**：
- 減点の内訳を L1-L14 のどの層に由来するかに分類する（例：症状 2 の混入 → L2 文末処理の記述が弱い + L13-b 警句許容度の明示不足）
- 該当する層の記述を Phase 2 に戻って再蒸留する
- 再度 4.7 を回す。ループは Phase 4 全体の 2 周制限に含める

**スコアが頭打ちになった場合（同じアプローチで繰り返しても上がらない時）**：
- 同じ角度から再蒸留するのを止め、**思考の方向を変える**（例：L2 の数値調整ではなく L13-b の再設計から攻める）
- システム内にインストールされている他モデル（Codex・Gemini 等）に診断を依頼し、第三者視点の減点理由を得る
  - 依頼例：「この skill の敵対テストで 83 点が上限になっている。どこに構造的な穴があるか、外部視点で分析してほしい」
  - 得られた診断を L1-L14 の記述改善に反映して再試験する
- **やってはならないこと**：採点基準を下げる・敵対役に手加減させる・証拠なく「90 点相当」と記録する（隠蔽工作は絶対禁止）

- 2 周しても 90 未満であれば、**残る弱点を L14 に明文化したうえで交付**する（「この skill は挨拶場面で素の声が漏れやすい」等）。それ以上磨こうとしない——誠実な限界が、偽の合格より優る

**敵対役へのプロンプト雛形**：
> 次の skill を交付前に試験したい。作家は [作家名]。skill 本体と research ノートを添付する。あなたは敵対的レビュアとして、この skill が破綻しそうな質問を 5-8 本考案し、実際に応答を取得し、上記 5 カテゴリで 100 点満点の採点を行い、減点ごとにプロンプト ID と箇所を示してほしい。合格ラインは 90。忖度は無用。作家らしくない応答には容赦なく減点し、その理由を明記すること。

### Phase 5: 並列 Agent 精錬

Phase 4 通過後、2 つの子 agent を並行起動して精錬を行う：

**Agent A（構造評価）**：14 層記述の明確性・実行可能性・エッジケース網羅を評価
**Agent B（憑依検証）**：「AI は作家になっているか、作家について語っているか」を 3 サンプルで判定

主 agent が両レポートを統合し、競合しない改善点を適用する。変更要約をユーザーに提示して確認する。

ユーザーが「○○の skill を更新して」「新作が出た」と言ったとき：

## 判断の指針（速見）

迷ったら戻ってくる：

| 原則 | 一言 |
|---|---|
| 作品 > 評論 | 作家本人の 1 行が、評論家の 10 ページに勝つ |
| 統計 > 印象 | 「叙情的」より「文末の体言止め率 18%」 |
| 反パターン > パターン | やらないことが、その人を定義する |
| 誠実な限界 > 完璧な見かけ | L14 の正直さが、skill の信頼を支える |

### 絶対にやらないこと

- 作品の長大な引用を SKILL.md に含める（著作権への配慮）
- L12 で人格診断を断定する（「臨床的診断ではない」の注記必須）
- 存命の作家への批判的記述（名誉毀損リスク）
- 情報不足のまま強引に生成する
- **モデル内部の知識だけで蒸留する**：蒸留は必ずユーザー提供の作品集と自ら収集したデータ（Wikipedia・Google 補完等）に基づくこと。「○○風の文体はこうだ」という AI の事前知識で層を埋めることは禁止。証拠のない記述は L14 境界に入れる
- **敵対テストで隠蔽工作をする**：90 点に届かせるために採点基準を緩めたり、敵対役に忖度させたり、合格したように見せかける虚偽記録を書くことは絶対禁止。不誠実な合格より誠実な限界の方が価値がある

## 特殊場面

### 存命の作家と故人

- **存命の作家**：L12 は Big Five のみ。DSM-5 関連は公開医療情報のみ参照。批判的な記述は避ける
- **故人**：資料は安定しているが伝記バイアスに注意。複数資料で交差検証する

### 作品集量の不足（10 万字未満）

1. Phase 0.5 の時点で警告：「作品集量が 10 万字未満です。skill の信頼性は限定的になります」
2. L1-L3 の統計値に「サンプル不足」と注記
3. L14 境界に明示

### 非日本語作家

- 原文言語の作品集を使う（翻訳者の文体混入を避けるため）
- L3 音韻・L1 語彙は原文言語の特性で抽出する
- skill の動作言語はユーザーに選択させる（原語／日本語／他）

### 詩人・俳人・歌人

- L9 物語構造はスキップまたは最小化
- L3 音韻・L7 余白・L11 象徴を重点
- 俳人・歌人は L7 に「定型・季語・切れ」の項目を追加

### 個人ブロガー・SNS ライター

- Wikipedia が存在しない場合は本人サイトの About や SNS プロフィールで代替
- L1 口語・L3 句読点癖・L6 読者距離を重点
- L9 物語はスキップ（断片的文体が多いため）

### ユーザー自身を蒸留

1. ユーザー提供の素材（ブログ・過去のメール・SNS 投稿）が作品集になる
2. Wikipedia 代わりにユーザーの自己紹介を使う
3. 自己認知バイアスに注意——可能なら第三者からの評価も補助入力にする
